## 0. 議論の順番 ( 発散させないための型 )

1. 現状を数値で診断する ( いまどのバランスか )

5. 編集ルール ( 採否基準 ) は評価指標から従属的に定まる

## 2. 中心構造: ユーザージャーニー × ソース種別

### 4 ペルソナのジャーニーが必要とするソース

| ジャーニー段階 | 必要なソース種別 | 現状 |
|---|---|---|
| 知る ( 現状把握 ) | 一次資料、統計、白書 | ○ 充足 |
| 着想 ( アイデア生成 ) | 道内 / 道外取り組み事例、構造分析 | △ 道外不足 |
| 実行 ( 行動に移す ) | 制度ガイド、補助金一覧、連絡先、How-To | × 大幅に手薄 |

> **論点**: 現状は「知る」に偏重。「実行」を支えるソースが圧倒的に少ない。協力隊・移住希望者・起業家が「次の一歩」を踏むためのリソース ( 補助金、相談窓口、制度詳細 ) を増やす必要がある。

## 3. ソース拡大の方法 ( 5つの方法論 )

### A. プッシュ型 ( 自動で流れ込む )

- 自治体 RSS / 公報フィード
- 報道メディアの RSS / API ( 北海道新聞、NHK、日経 )
- 大学リポジトリ ( CiNii、JAIRO Cloud )
- ふるさと納税プロジェクトの公開データ

**利点**: 一度仕掛ければ持続。
**欠点**: ノイズが多い、フィルタリング設計が要る。

### B. プル型 ( 能動的に取りに行く )

- 自動リサーチルーチン ( 既存、Sun/Wed 稼働 )
- WebSearch による定期クエリ ( キーワード × 北海道 × 年 )
- API 直接叩き ( CF プラットフォーム、政府オープンデータ )

**利点**: 狙ったテーマを集められる。
**欠点**: キーワード設計の質に依存。

### C. キュレーション型 ( 人が見つけて投入 )

- 大東 個人の Twitter / X 観測
- 業界ネットワーク経由 ( キメラ、Largo、北海道関係者 )
- 専門家への取材・推薦

**利点**: 質の高い情報、文脈つき。
**欠点**: スケールしない、属人化。

### D. 連携型 ( 組織と協定 )

- 北海道新聞社 / 道北ネット ( 既に大東経由 )
- 道庁・市町村との情報提供協定
- 大学・シンクタンクとのデータ共有

**利点**: 質と量の両立、独占的アクセス。
**欠点**: 立ち上げが重い、関係構築に時間。

### E. ユーザー寄与型 ( 読者から )

- 「掲載依頼フォーム」設置
- GitHub Pull Request ( 既にできる構造 )
- コメント機能で出典報告

**利点**: 現場の最新情報、コミュニティ感。
**欠点**: 質のばらつき、運用負荷。

**推奨ミックス**: B + C を主軸、A を計画的に追加、D は中期、E は長期 ( 信頼性レビュー体制ができてから )。

## 4. 評価指標 ( ここが議論の核心 )

ソースを掲載するかどうか、また UI でどう見せるかの判断軸。

### 4つの一次指標

| 指標 | 定義 | 評価方法 | 表示 |
|---|---|---|---|
| **信頼性** | 一次ソース性 + 主体権威 | 一次 / 二次 + 主体カテゴリ | バッジ ( 一次 / 報道 / 研究等 ) |
| **鮮度** | 公表日からの経過 | year フィールド + 4段階分類 ( 既存 ) | 色付き年バッジ ( 既存 ) |
| **関連性** | ISSUE テーマとの一致 | 編集判断 1-5 | 内部のみ ( UI 非表示 ) |
| **持続性** | 5+ 年継続 or 構造的 | 編集判断 ( 継続的 / 単発 ) | バッジ ( 「長期継続」等 ) |

### 4つの二次指標 ( 補助 )

- アクセス性 ( 無料 / 有料 / 言語 )
- 引用可能性 ( DOI 有無、安定 URL )
- データ提供 ( 元データダウンロード可 )
- 言語 ( 日本語 / 英語 / その他 )

### スコアの集約

**ジャーニー貢献度**: 各ソースが「知る / 着想 / 実行」のどれを支えるかをタグ付け ( 1 〜 複数 )。フィルタや UI 強調で「実行」のための情報が必要なユーザーが辿りやすくする。

> **論点**: 内部スコアを全て UI 表示するか、バッジ簡略表示にとどめるか。情報を増やすと読み手の認知負荷も増える。バッジ 2-3種類 ( 一次 / 鮮度 / 長期継続 ) に絞るのが現実的。

## 5. 編集ルール ( 評価指標が決まれば従属的に定まる )

評価指標を 4つ決めた前提で、採否基準:

### 必須 ( これを満たさないと載せない )
- URL 実在、現行で 200 を返す
- 出典主体が明示できる ( 道庁、〇〇大学等 )
- ISSUE テーマと直接関連する
- 既存 SOURCES と重複しない

### 推奨 ( 満たすほど優先 )
- year ≥ 2024 ( 鮮度 )
- 一次ソース ( 信頼性 )
- 構造分析 or 5+ 年継続 ( 持続性 )
- ジャーニー「実行」段階を支える ( 補助金、制度 )

### 禁止 ( 載せない )
- 個人ブログ ( ただし著名な専門家除く )
- 短期キャンペーン記事 ( バズ目的 )
- 古い情報のみ ( ≤ 2018、歴史的価値ある場合除く )
- 偏った主張 ( 編集判断 )

## 6. ロードマップ ( 短 / 中 / 長 )

### 短期 ( 〜 1ヶ月 )
- [ ] SOURCES スキーマに `tier` ( 一次 / 二次 ) と `journey` ( 知る / 着想 / 実行 ) を追加
- [ ] UI: ソースバッジ表示 ( 信頼性 + 鮮度 + 長期継続 )
- [ ] フィルタ: 「実行を支える」ソースのみ表示
- [ ] 既存 60件の SOURCES に tier / journey を backfill ( 私が手動 )
- [ ] 自動ルーチンに評価指標を反映 ( prompt 更新 )

### 中期 ( 〜 3ヶ月 )
- [ ] プッシュ型ソース追加: 北海道新聞 RSS、道庁プレスフィード
- [ ] CF API による プロジェクトデータ自動取得
- [ ] 大学リポジトリからの研究取り込み ( CiNii API )
- [ ] 海外事例 ( externalCases ) の拡充 ( 国際比較ソース 10件 )

### 長期 ( 〜 1年 )
- [ ] 連携: 北海道新聞・道庁との情報提供協定
- [ ] ユーザー寄与: 掲載依頼フォーム + PR レビュー体制
- [ ] 専門家への定期取材 ( 月 1-2回 )
- [ ] 編集委員会の組成 ( 評価判断の客観性確保 )

## 付録: 既存システムとの整合

- 既存スキーマ ( t, o, u, year ) は そのまま、`tier` / `journey` を optional 追加
- 既存の鮮度バッジ ( fresh / mid / stale / unknown ) は維持
- 自動ルーチンのプロンプトに評価指標条件を追加
- ユーザー UI への影響は最小化 ( バッジ追加程度 )
