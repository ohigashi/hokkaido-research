# 1人とClaudeで「北海道・地域課題リサーチ」を24時間で12段階改修した話

URL: https://hokkaido-research.lrg.jp/

---

ローンチしたのは2026年6月12日。北海道の地域課題を構造から読み解いて、道内・道外の長期事例から学べる読み物プラットフォームです。

24時間後の今、ISSUEは20→66件、記事は30→104本、12段階の機能改修を入れて、ぼくも自分でも何が起きたのかよくわかっていません。これはその24時間の開発ストーリー。AIを相棒に1人で動かすと、こんな速度で何かが組み上がるんだなあ、という記録です。

---

## なぜ作ったか

道内人口は2025年の国勢調査速報で498万人。70年ぶりに500万人を割り込みました。道内合計特殊出生率は1.01で全国45位。札幌市は0.96で全国主要市の最低水準。

道で起きていることを「ぼくたちは知らなさすぎる」というのが、始まりの感覚でした。道内外で何が起きていて、どこの町がどんな手を打って、何が残ったのか。それを構造で読み解く媒体が欲しかった。

ベンチマークにしたのは、海士町・神山町・上士幌町といった「長期に構造を組み替えた」町の事例。単発の話題ではなく、5年10年続けて何が町に残ったか、を見る軸を媒体全体に通したかった。

---

## ローンチ直後の自分への違和感

公開してすぐ、自分で読んで思いました。中身が薄い。

ISSUEは形になっていても、記事のほとんどは「数値で見る現状→5つの実装→道外事例→わたしたちにできること」のテンプレ。5年以上継続した事例の重み、推移の説得力、ファクトと解釈の区別、全部が足りない。

そこから24時間、ずっと組み直しています。

---

## Phase別の改修ログ

### Phase 1-3 : 編集基盤を厚くする

ISSUEを20→66件に拡張。用語集 ( 24→65用語 ) を組んで、記事本文中の用語を初出のみ自動でリンク化。半角スペース・中黒前後の空白を2,700箇所くらい一気にクリーンナップ。

文字コードと記号は地味でつまらないけれど、ここを揃えないと「読み物」にならない。

### Phase 4 : 品質強化サイクル

記事1本ずつを8軸 ( 文字数 / 表 / 具体数値 / 事例の固有名 / 出典 / 実行アクション / 中心問い / 残る資産 ) で採点するスクリプトを作りました。全104記事を採点→弱い記事をリスト化→Notionに強化キューとして蓄積する仕組み。

ここからは「自動で強化候補を選ぶ→ぼくがレビュー→反映」のループが回るようになります。

### Phase 5 : スコアアルゴリズム

道に向けて発信するなら「質の高い記事を露出する」をシステムで保証したい。そこで設計したのがcompositeスコア。

- 構造スコア ( 0-100 ): 文字数・表・出典など
- GA4エンゲージメントスコア ( 0-25 ): 内部回遊PV×平均滞在秒
- 打席ブーストスコア ( 0-15 ): リライト後21日間の試運転露出
- 鮮度はスコアに加算せず、同点時のtiebreaker

質の定義はこう決めました。「内部回遊でよく読まれて、エンゲージメントも強い記事 = 質が高い」。外部SNSでバズって流入が多くても、読まれていなければTOPには出さない。流入は記事 ( 入口 ) 、回遊はTOPと関連記事 ( ループ ) 、提供価値はその両方で立ち上がる、という整理。

### Phase 6 : 回遊強化

TOPの読み物セクションは、表示中の課題フィルタと連動させました。ふるさと納税を絞り込んでいたら、ふるさと納税関連の記事が上に出る。記事下の関連記事も、共通ISSUE数+品質+GA4+鮮度の合成スコアで選定。

### Phase 7 : ファクト・仮説・推測の区別

書きながら一番苦労したのが、ぼくの解釈と事実の混ざり具合。1つの段落で「○○は××であり、これは△△を意味する」と書くと、どこまでが一次資料の事実で、どこからがぼくの見立てなのか読者に伝わらない。

なので3種類のバッジを作りました。

- FACT: 一次資料の事実・出典必須
- 仮説: 編集部の解釈・因果のつなぎ
- 推測: 不確実だが想定される将来展開

書くたびに自分で「これは本当に出典が引けるか?」「これは解釈なのに事実みたいに書いていないか?」を問い直すようになります。読者にとっても、編集者にとっても、たぶん良い装置。

### Phase 8 : フィルタをTOPと同役割同デザインで

`/articles/` 一覧に、おすすめ/新しい順/アップデート順の3ソートと、8課題分類・地域・地域特性・キーワード ( 自動5件+フリーワード ) のフィルタを実装。TOPの課題カードと同じデザイン哲学で踏襲しています。

### Phase 9 : つぶやき編集フロー

これが個人的に一番うれしい仕組み。

日々の取材・インタビュー・思いつきを、スマホから音声入力でNotion DBに投げ込めるようにしました。写真や音声はGoogle Drive専用フォルダにアップ。ぼくがClaudeに「つぶやきメモ取得」と声をかけると、未処理メモを順番に判断 ( 新規記事/既存リライト/ISSUE更新/スルー ) してくれる。

つまり、取材で得た一次情報がそのまま媒体の編集インボックスに流れ込む構造。外出先からスマホで思いついたまま吐き出して、帰ってきたら整理する。

### Phase 10-11 : 全面リライト10本

これがいちばんしんどかった。主要10本を全面リライトしました。

道人口498万・70年ぶり500万割れ。ラピダス政府支援累計2.9兆円。JR北海道営業赤字582億円。道内秋サケ漁獲量14.4→5.0万トン。道内合計特殊出生率1.27→1.01。こういう推移を一次資料で取り直して、道外事例も海士町・神山町・上士幌・流山・明石・Buurtzorg・デンマーク・ノルウェーまで突っ込んで構造を組み直しています。

データはスナップショットではなく推移で示す、というのもここで決めた方針。「2024年X万人」だけでは構造は見えなくて、「2014年→2019年→2024年でX→Y→Z」と並べると、速度と方向が見える。

### Phase 12 : パイプライン整備

`refresh_quality.sh` という1コマンドで、「記事生成→GA4取得→採点→スコア注入→記事再生成→git push」が走ります。リライトしたら自動で推定スコアが付いて、21日間の打席ブーストでTOPに上がるループ。

実データが溜まったら推定値が実値に置き換わる設計。スタート地点を推定値で作っておくのがポイントで、待たずに動かせる。

### Phase 13 : ビジネスアイデアを各課題に

直近で足したのが、ISSUEと記事の両方に「ビジネスアイデア」セクション。課題から逆算した新規事業・起業・投資の方向性を1-3個ずつ。ターゲット・仕組み・収益モデル・組み合わせを並べて、読者が「自分ならどう動くか」を考える出発点にする。

## 思想として置いたこと

3つだけ。

1. 主題は道内ケース・補強は道内外問わず広く参照。「道外事例の紹介記事」化を避ける
2. 新しい≠良い。古くても内部回遊でよく読まれてエンゲージメントが強ければ質が高い
3. 記事は流入の起点+提供価値の両立。SEOとコンテンツ質を同時に走らせる

ここを軸にしてあれば、ぼくが眠っている間にClaudeが品質強化サイクルを回しても、ブレない。

## やってみて思ったこと

1人とAIで動かすと、速度が変わるというよりは「仕組みを直して、仕組みに任せて、また直す」のサイクルが回せるようになる、という感覚が強かったです。

ぼくが書くより速く、ぼくの判断より広く、でもぼくの編集方針からは外れない。そういう状態を作るのは、AIが賢いから、ではなく、「方針をメモリに保存して、そのメモリを毎回参照する仕組み」が成立しているからだと思う。

つぶやきメモから記事になるサイクルがちゃんと回り始めたら、たぶん次の景色が見えてくる。とりあえずぼくは取材に出ます。
